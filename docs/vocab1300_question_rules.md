## 1. このルールの目的

このファイルは、青のまなびクエスト v1 で使用する「語彙力UP1300」の問題作成ルールをまとめたものです。

v1では、語彙力UP1300をもとに、青ちゃんが語彙を忘却曲線に沿って復習できるWebアプリを作成します。

ただし、教材本文や問題文をそのまま使うのではなく、語句・意味・問題形式を参考にしながら、青ちゃん向けのオリジナル問題として作り直します。

---

## 2. 前提

### 対象

- 学習者：青ちゃん
- 学年：小学3年生
- 教材：語彙力UP1300
- ツール名：青のまなびクエスト
- 管理名：ao-learning-quest

### v1の学習目標

語彙力UP1300の1300語のうち、90%を「定着○」にする。

### 問題作成担当

問題作成はGPが担当する。

問題作成スレッドでは、このルールを前提に、スキャンPDFから語句を確認し、オリジナル問題を作成する。

---

## 3. 作問の基本方針

### 最重要ルール

教材の本文・例文・選択肢・解説をそのまま使わない。

語句、読み、意味、問題形式を参考にして、青ちゃん向けに新しく作る。

### 問題の基本形

基本は次の形にする。

1. 短い文脈、例文、ヒント、またはイラスト
2. 問い
3. 4択
4. 正解
5. 短い解説
6. JSON形式

### 青ちゃん向けの文章レベル

- 小学3年生が読める長さにする
- 文はできるだけ短くする
- 1問につき、例文は1〜2文を基本にする
- 難しい漢字を使う場合は、必要に応じてひらがなにする
- 語句そのものは漢字で出してよい
- 説明はやさしくする
- ただし、幼児向けにしすぎない

### 例文の場面

青ちゃんが想像しやすい生活場面・学校場面・遊び場面に置き換える。

使いやすい場面の例：

- 発表
- けんか
- 友だちとの会話
- 先生とのやりとり
- 家族との会話
- 習い事
- 遊び
- 読書
- ぬいぐるみ
- 試合
- おもちゃ
- なくし物

## 4. 問題タイプ

v1では、次の6タイプを基本とする。

| type | 内容 | 形式 | イラスト |
|---|---|---|---|
| `meaning_context` | 例文中の語句の意味 | 短い文脈＋意味4択 | 不要 |
| `reading_context` | 読み方で意味が変わる語 | 例文＋読み方4択 | 不要 |
| `age_word` | 年齢を表す特別な語 | 読み方4択／何歳か4択／由来ヒント4択 | 不要 |
| `image_word` | 絵を見て場面・語句を選ぶ | イラスト＋言葉4択。必要に応じて `hint` を入れる | 必要 |
| `month_hint` | 月の異名 | ヒント文＋何月か4択。`subtype: "month_word"` を推奨 | 任意 |
| `counter_image` | 物の数え方 | イラスト＋数え方4択。`counterTarget` / `counter` / `counterReading` を推奨 | 必要 |

## 5. 共通の作問ルール

### 5.1 問題文

- 文脈から語句の意味を考えられるようにする
- 問題文だけで正解を推測できるようにしすぎない
- ただし、知らないと絶対に解けない問題にも寄せすぎない
- 青ちゃんが読んで状況を想像できる文にする
- 家庭、学校、友達、習い事、読書、遊びなど、身近な場面を使う

### 5.2 選択肢

- 選択肢は4つ
- 正解は必ず1つ
- 選択肢の長さはできるだけそろえる
- 正解だけが不自然に長くならないようにする
- 誤答は完全に変すぎるものにしない
- ただし、ひっかけすぎない
- 小3が読める日本語にする
- 似た意味が重なりすぎる選択肢は避ける

### 5.3 正解位置とシャッフル

正解位置は、作問データ上では極端な偏りを避ける程度でよい。

ただし、アプリ表示時には選択肢をランダムにシャッフルする。  
そのため、JSON内の `answerIndex` は元データ上の正解位置として扱い、画面表示時にはシャッフル後の正解位置をアプリ側で再計算する。

青ちゃんが「答えはだいたいア」などと覚えないように、実際の出題画面では正解位置が固定されない仕様にする。

#### 作問時の扱い

- JSONには `choices` と `answerIndex` を入れる
- `answerIndex` は、JSON内の `choices` 配列における正解位置
- `answerIndex` は0始まり
  - ア = 0
  - イ = 1
  - ウ = 2
  - エ = 3
- 作問時点では、正解位置を完全に均等にしなくてよい
- ただし、同じページ内で極端に偏る場合は見直す

#### アプリ表示時の扱い

- 問題表示時に `choices` をランダムにシャッフルする
- 元の `answerIndex` から正解の選択肢テキストを特定する
- シャッフル後の配列に合わせて、正解位置を再計算する
- 回答判定は、シャッフル後の正解位置で行う

#### answerIndexCheck

`answerIndexCheck` は、作問データ上で正解位置が極端に偏っていないかを確認するために使う。

ただし、アプリ側で選択肢をシャッフルするため、ア・イ・ウ・エを完全に均等にする必要はない。

例：

- 6問で `2・2・1・1` 程度なら問題なし
- 6問で `ア4問、イ0問、ウ2問、エ0問` のような偏りは見直す
- 途中まで作成済みのページに問題を追加する場合、既存の `answerIndex` を参考にし、不足している位置を優先してもよい

### 5.4 解説

- 解説は1〜2文
- 青ちゃんが読んで意味がわかる文にする
- 正解の意味を確認する
- 必要に応じて「〜という意味ではありません」と誤解を防ぐ
- 長い文法説明はしない
- よい意味にも悪い意味にも使われる語は、基本の意味を説明したあと、一文だけ補足してよい

### 5.5 表記

- 問題内では、対象語句を太字にしてもよい
- JSON内ではHTMLタグは使わない
- アプリ表示側で強調する
- 句読点は自然につける
- 選択肢の先頭に「ア」「イ」「ウ」「エ」はJSONには入れない
- JSONでは選択肢配列のみを持つ
- JSONとして保存する場合は、引用符は必ず半角の `"` を使う

### 5.6 青ちゃんの名前の使用

例文に「青ちゃん」を使ってよい。

ただし、毎問使うと不自然になるため、同じページ内では1〜2問程度までを目安にする。

青ちゃん以外にも、次のような表現を使って場面に変化をつける。

- わたし
- 友だち
- 先生
- 妹
- 家族
- チーム
- 相手
- その人

### 5.7 長い語句・慣用句の扱い

「居ても立ってもいられない」のような長い語句は、分解せずに `word` にそのまま入れる。

ただし、アプリ表示では長くなるため、長い語句は問題画面で折り返し表示できる前提にする。

### 5.8 subtype の扱い

必要に応じて、問題タイプとは別に `subtype` を入れてよい。

`subtype` は、問題形式ではなく、語句カテゴリを表す補助分類とする。

例：

```json
{
  "type": "image_word",
  "subtype": "manner_word"
}
```

v1では `subtype` は任意項目とする。  
ただし、p014のように「気づかい・マナー・しぐさ」を表す語は、原則として `subtype: "manner_word"` を入れる。


### 5.9 p011〜p025で追加された補足ルール

#### 打ち消しとセットになりやすい語

「さほど」「過ぎない」のように、特定の文型や打ち消し表現と一緒に使われやすい語は、例文の中で自然な形で使う。

解説には、必要に応じて使い方を一文だけ補足してよい。

例：

```txt
さほどは、それほど、たいして、という意味です。多くの場合、後ろに打ち消しの言葉が続きます。
```

#### 副詞系語句

「そくざに」「いささか」「えてして」「さしずめ」のような副詞系語句は、短い例文だけでは意味が取りにくいことがある。

そのため、解説では必ずやさしい言い換えを入れる。

例：

```txt
えてしては、とかく、大体の場合において、という意味です。
```

#### 抽象語

「概念」「形式」「発揮」「秩序」「真相」「正常」などの抽象語は、青ちゃんが想像しやすい場面に寄せる。

使いやすい場面例：

- 学校
- 授業
- 発表
- 試合
- 図書室
- 友だちとの会話
- 家族との会話

抽象語の選択肢は、意味の粒度をそろえつつ、正解が明確に1つになるようにする。

#### 似た意味・近いカテゴリの語句

同じページに意味が近い語句がある場合、誤答に使いすぎると混乱しやすい。

例：

- 願ったりかなったり
- 願ってもない

このような近い語は、必要以上に互いの誤答として使わない。使う場合は、例文と解説で違いが分かるようにする。

#### ひらがな語句

「うやうやしい」「えんえん」「いささか」「えてして」のように語句自体がひらがなの場合も、`reading` には同じ表記を入れる。

```json
{
  "word": "えてして",
  "reading": "えてして"
}
```

#### 慣用句の読み

「歯を食いしばる」「得体の知れない」「借りてきた猫のよう」のような慣用句は、`word` には漢字かな交じり、`reading` には全文ひらがなを入れる。

```json
{
  "word": "歯を食いしばる",
  "reading": "はをくいしばる"
}
```

#### 体の動作を含む慣用句

「首を縦にふる」「手招き」「眉をひそめる」「ほおをふくらます」のような体の動作を含む語句は、将来的に `image_word` 化できる。

ただし、画像生成枚数が増えすぎるため、v1では基本的に `meaning_context` として扱う。

## 6. 誤答選択肢の作り方

同じページに出てくる別語句の意味を、誤答選択肢として使ってよい。

これはページ内の語句をまとめて復習しやすくするために有効です。

ただし、次の点に注意する。

- 正解と意味が近すぎるものは避ける
- 誤答だけ明らかに不自然にならないようにする
- 同じ誤答ばかり使い回さない
- 正解だけが具体的で、誤答だけが抽象的になりすぎないようにする
- 誤答が別の問題の正解になる場合でも、その問題では明確に誤答になるようにする

---

## 7. 抽出データ `extractedWords`

問題JSONとは別に、PDFから抽出した語句データを `extractedWords` として持つ。

これは作問前の整理データであり、アプリ本体に必ず入れるデータではない。

### 基本形

```json
{
  "sourcePage": 12,
  "extractedWords": [
    {
      "word": "口下手",
      "reading": "くちべた",
      "meaning": "話すことがあまり上手ではないこと",
      "detectedType": "meaning_context"
    }
  ]
}
```

### extractedWords の項目

| 項目 | 内容 |
|---|---|
| `word` | 対象語句 |
| `reading` | 読み |
| `meaning` | 青ちゃん向けに整理した意味 |
| `detectedType` | 問題タイプ |

## 8. type別ルール

## 8.1 `meaning_context`

### 用途

語句の意味を、短い文脈の中で考える問題。

### 形式

- 1〜2文の例文
- 語句の意味を問う
- 4択
- 解説

### 問いの例

- 「口下手」の意味はどれ？
- 「客観的」の意味はどれ？
- 「陰口」の意味はどれ？

### 作問ルール

- 例文は教材のものをそのまま使わない
- 語句が自然に使われる場面を作る
- 文脈で意味が少し想像できるようにする
- 語句の使い方も学べるようにする
- ひとつの問題に複数の新しい語句を入れすぎない

### よい例

青ちゃんは、言いたいことはあるのに、みんなの前ではうまく話せませんでした。  
青ちゃんは少し口下手です。

口下手 の意味はどれ？

### 注意

- 例文が長くなりすぎないようにする
- 文脈だけで正解が明らかすぎる場合は少し調整する
- 誤答選択肢に同じページの別語句の意味を使ってもよいが、混乱しすぎないようにする


## 8.1.1 `meaning_context` + `subtype: "onomatopoeia_word"`

### 用途

「ごうごう」「さんさん」「とうとう」「まんまん」「こうこう」「せつせつ」など、音・光・動き・様子・気持ちを表すオノマトペを扱う。

v1では、オノマトペも基本的には `meaning_context` として扱う。

ただし、管理しやすくするため、必要に応じて `subtype: "onomatopoeia_word"` を付ける。

```json
{
  "type": "meaning_context",
  "subtype": "onomatopoeia_word",
  "word": "ごうごう",
  "reading": "ごうごう"
}
```

### 出題方針

オノマトペは、文脈の中で意味を選ぶ問題にする。

例：

```json
{
  "sentence": "台風の日、外では風がごうごうと鳴っていました。",
  "question": "ごうごう の意味はどれ？"
}
```

### 解説方針

解説では、音・光・動き・気持ちなど、何の様子を表しているのかを短く説明する。

例：

```txt
ごうごうは、大きな音が響いて、騒がしいようすです。
```

### 選択肢の注意

同じ系統のオノマトペがある場合は、正解と誤答が近くなりすぎないように注意する。

例：

- `さんさん`：太陽が光り輝くようす
- `こうこう`：明るく、きらきらと輝くようす

どちらも「光る」系の語なので、例文や解説で違いが分かるようにする。

### 画像化について

オノマトペは画像化できるものもあるが、v1では画像問題にしすぎない。

基本は `meaning_context` として作問する。画像があると明らかに理解しやすい場合のみ、将来的に `image_word` 化を検討する。

例：

- `ごうごう`：強い風や川の音
- `さんさん`：太陽の光
- `とうとう`：川の流れ

ただし、画像生成枚数が増えすぎるため、v1では原則として画像なしでよい。

## 8.1.2 `meaning_context` + `subtype: "animal_idiom_word"`

### 用途

動物を含むことわざ・慣用句を扱う問題。

例：

- 犬も食わない
- 馬が合う
- 閑古鳥が鳴く
- 猿も木から落ちる
- おうむ返し
- 猫の手も借りたい

v1では基本的に `meaning_context` として扱い、管理用に `subtype: "animal_idiom_word"` を付ける。

```json
{
  "type": "meaning_context",
  "subtype": "animal_idiom_word"
}
```

### 出題方針

- 動物そのものの意味を問うのではなく、ことわざ・慣用句としての意味を問う
- 例文では、比喩としての意味が伝わる場面にする
- 必要に応じて解説で「たとえ」「実際の動物の話ではない」ことが分かるようにする

例：

```json
{
  "sentence": "長い練習をしてきたプロでも、大事な試合で失敗することがあります。猿も木から落ちることがあるのです。",
  "explanation": "猿も木から落ちるは、どんなに得意なことでも失敗することがある、というたとえです。本物の猿の話ではありません。"
}
```

## 8.1.3 `meaning_context` + `subtype: "period_word"`

### 用途

時期・期間を表す語句を扱う問題。

例：

- 上旬
- 中旬
- 下旬
- 年始
- 年末

v1では基本的に `meaning_context` として扱い、管理用に `subtype: "period_word"` を付ける。

```json
{
  "type": "meaning_context",
  "subtype": "period_word"
}
```

### 出題方針

- 「いつごろか」「どの期間か」が分かるように、例文や解説で具体的に示す

各語の目安：

- 上旬：月の初めのころ。およそ1日〜10日。
- 中旬：月の真ん中のころ。およそ11日〜20日。
- 下旬：月の終わりのころ。およそ21日〜月末。
- 年始：年の初め。
- 年末：年の終わり。

### 注意

月の異名を問う `month_hint` / `month_word` とは分ける。

`period_word` は、月名そのものではなく、時期・期間を表す語句に使う。

## 8.2 `reading_context`

### 用途

同じ漢字でも読み方によって意味が変わる語を学ぶ問題。

例：

- 生地
- 辛い
- 末期

### 形式

- 例文を出す
- 対象語句の読み方を問う
- 4択
- 解説

### 問いの例

次の文の「生地」は、何と読む？

赤と黒の生地を、それぞれ買う。

### 作問ルール

- 文脈で読み方が決まる例文にする
- 読み方の選択肢はひらがなで出す
- ありえない読み方だけにしない
- 読み方と意味を解説で短く確認する

### JSONの注意

`word` には漢字表記を入れる。  
`reading` には正しい読みを入れる。  
`choices` には読み方候補を入れる。

### `age_word` との使い分け

`reading_context` は、同じ漢字でも文脈によって読み方や意味が変わる語を扱う。

一方で、不惑・還暦・喜寿・卒寿・白寿のような「年齢を表す特別な語」は、読み方を問う問題であっても、分類上は `age_word` とする。

例：

- `reading_context`：生地、辛い、末期など
- `age_word`：不惑、還暦、喜寿、卒寿、白寿など

## 8.3 `age_word`

### 用途

年齢を表す特別な語を学ぶ問題。

例：

- 不惑
- 還暦
- 喜寿
- 傘寿
- 米寿
- 卒寿
- 白寿

### 基本方針

`age_word` は、「読み方」「何歳を表すか」「漢字や言葉の由来」をセットで覚える語を扱う。

p013のような年齢語は、読み方を問う問題であっても `reading_context` ではなく、原則として `age_word` とする。

### 形式

v1では、まず読み方を問う問題を基本にする。

- 文の中に年齢語を出す
- その語の読み方を問う
- 4択
- 解説で「読み方」と「何歳を表すか」を確認する
- 由来が覚えやすい語は、解説に一文だけ由来を入れる

### 問いの例

文の中の 還暦 の読み方はどれ？

### 解説の例

還暦は「かんれき」と読みます。六十歳を表す言葉です。

由来が覚えやすい語の場合：

喜寿は「きじゅ」と読みます。七十七歳を表す言葉です。「喜」の字をくずすと七十七に見えることからきています。

### 作問ルール

- 読み方を問う問題では、選択肢はひらがなで出す
- 解説には必ず「読み方」と「何歳を表すか」を入れる
- 由来が覚えやすい語は、一文だけ由来を補足する
- 数え年と満年齢の細かい違いは、通常問題では深く扱わない
- 青ちゃんが混乱しやすい場合は、「何歳を表す言葉です」と簡潔に説明する

### 誤答選択肢

`age_word` の読み方問題では、完全にありえない読みだけでなく、青ちゃんが選びそうな読み間違いを入れる。

誤答の作り方の例：

- 音読みの違い
- 訓読みの混入
- 長音化
- 小さい「っ」の混入
- 似た音の読み

ただし、不自然な読みばかりにすると簡単になりすぎるため、自然そうな誤読を1〜2個入れる。

### JSONの注意

正式な問題データでは、`type` は `age_word` とする。

```json
{
  "type": "age_word",
  "word": "喜寿",
  "reading": "きじゅ",
  "sentence": "おばあちゃんの喜寿のお祝いに、みんなで手紙を書きました。",
  "question": "文の中の 喜寿 の読み方はどれ？",
  "choices": ["きことぶき", "よろこびじゅ", "きじゅう", "きじゅ"],
  "answerIndex": 3,
  "explanation": "喜寿は「きじゅ」と読みます。七十七歳を表す言葉です。「喜」の字をくずすと七十七に見えることからきています。"
}
```

`age_word` では、任意項目として `questionMode` を入れてよい。

読み方を問う問題では、次のようにする。

```json
"questionMode": "reading"
```

v1では `questionMode` は必須ではない。
ただし、今後「読み方」「何歳か」「由来ヒント」などを分けたくなった場合に備えて、入っていてもよい項目として扱う。

## 8.5 `month_hint`

### 用途

月の異名を覚える問題。

月の異名とは、睦月・弥生・葉月・文月・霜月・師走のように、昔の月の呼び名を表す語です。

例：

- 睦月
- 弥生
- 葉月
- 文月
- 霜月
- 師走

### 基本方針

p205のような月の異名は、原則として次の形で扱う。

```json
{
  "type": "month_hint",
  "subtype": "month_word",
  "month": 3
}
```

`month_hint` は問題形式、`month_word` は語句カテゴリとして使う。

`month` は任意項目だが、月の異名では入れることを推奨する。  
`month` の値は `"3月"` のような文字列ではなく、`1`〜`12` の数値で持つ。

### 形式

- 短い `hint`
- 「何月を表す言葉？」を問う
- 4択
- 解説

### 問いの例

弥生 は何月を表す言葉？

### 作問ルール

- 問題は「この月の異名は何月？」を基本にする
- 丸暗記だけでなく、由来や季節のヒントを少し入れる
- ただし、由来は難しくしすぎない
- `hint` は、季節や行事から月を思い出せる短文にする
- 画像は必須ではない
- 季節感を補助したい場合だけ、`image` に仮パスを入れてよい

### 解説ルール

`month_hint` の解説には、必ず次を入れる。

- 読み方
- 何月を表すか

由来が覚えやすい語は、一文だけ由来を補足する。

例：

```txt
霜月は『しもつき』と読み、11月を表します。霜が降りるころの月という意味からきたと言われています。
```

月の異名の由来には諸説あるものが多いため、解説では断定しすぎない。

よい表現：

- 「〜と言われています」
- 「〜という説があります」
- 「〜と関係があると言われています」

避けたい表現：

- 「必ず〜です」
- 「この由来だけが正しいです」

### 選択肢の作り方

選択肢は、できるだけ近い季節の月を混ぜる。

よい例：

- 弥生：1月・2月・3月・4月
- 文月：7月・8月・9月・10月
- 霜月：10月・11月・12月・1月
- 睦月：11月・12月・1月・2月
- 師走：9月・10月・11月・12月
- 葉月：7月・8月・9月・10月

まったく離れた月ばかりにしない方が、季節感で考える練習になる。

### JSONの例

```json
{
  "id": "vocab1300_p205_001",
  "source": "vocab1300",
  "sourcePage": 205,
  "type": "month_hint",
  "subtype": "month_word",
  "word": "弥生",
  "reading": "やよい",
  "month": 3,
  "hint": "草木が芽を出し、春らしくなってくるころの月です。",
  "question": "弥生 は何月を表す言葉？",
  "choices": ["1月", "2月", "3月", "4月"],
  "answerIndex": 2,
  "explanation": "弥生は『やよい』と読み、3月を表します。草木がいよいよ生いしげるころという意味からきたと言われています。",
  "image": null,
  "tags": ["vocab1300", "month_hint", "month_word", "p205"]
}
```

## 8.6 `counter_image`

### 用途

物の数え方を覚える問題。

例：

- 一膳
- 一串
- 一反
- 一輪

### 形式

- イラスト
- 「一（　）と数えます」
- 4択
- 解説

### 問いの例

おはしは、一（　）と数えます。

### 作問ルール

- イラストは必須
- 何を数えるかがひと目でわかるようにする
- 選択肢は助数詞だけにする
- 正解以外の助数詞も、学習済みまたはよくあるものにする
- 解説で「何をどう数えるか」を短く説明する

### 画像の注意

- 物だけを描く
- 余計な背景は入れない
- 文字・数字は入れない
- ファイル名は `images/counter_xxx.png` 形式にする

## 9. JSON仕様

### 基本形

```json
{
  "id": "vocab1300_p012_001",
  "source": "vocab1300",
  "sourcePage": 12,
  "type": "meaning_context",
  "word": "口下手",
  "reading": "くちべた",
  "sentence": "青ちゃんは、言いたいことはあるのに、みんなの前ではうまく話せませんでした。青ちゃんは少し口下手です。",
  "question": "口下手 の意味はどれ？",
  "choices": [
    "いろいろな見方で考えること",
    "一人で二つの役をすること",
    "話し方が上手ではないこと",
    "ねばり強くて手強いこと"
  ],
  "answerIndex": 2,
  "explanation": "口下手は、話すのがあまり上手ではないことです。言いたいことがない、という意味ではありません。",
  "image": null,
  "tags": ["vocab1300", "meaning_context"]
}
```

### 必須項目

| 項目 | 内容 |
|---|---|
| `id` | 一意の問題ID |
| `source` | 教材名。v1では `vocab1300` |
| `sourcePage` | 元ページ番号 |
| `type` | 問題タイプ |
| `subtype` | 任意。語句カテゴリ。例：`manner_word` |
| `word` | 対象語句 |
| `reading` | 読み。不明または不要なら空文字可 |
| `answerReading` | 任意。`reading_context` で使う正解の読み |
| `month` | 任意。月の異名などで使う。1〜12の数値 |
| `counterTarget` | 任意。`counter_image` で使う数える対象物 |
| `counter` | 任意。`counter_image` で使う助数詞のみ。例：`丁`、`足` |
| `counterReading` | 任意。`counter_image` で使う助数詞の読み。例：`ちょう`、`そく` |
| `sentence` | 例文。通常問題では必須。`image_word` では省略可 |
| `hint` | 任意。`image_word` では推奨。画像だけで意味が伝わりにくい場合の短い手がかり |
| `question` | 問い |
| `choices` | 選択肢4つ |
| `answerIndex` | JSON内の `choices` 配列における正解位置。0〜3 |
| `explanation` | 解説 |
| `image` | 画像パス。不要なら null |
| `tags` | タグ配列 |

### IDルール

基本：

```txt
vocab1300_p{ページ番号3桁}_{連番3桁}
```

例：

```txt
vocab1300_p012_001
vocab1300_p012_002
vocab1300_p205_001
```

### answerIndex の注意

`answerIndex` は、JSON内の `choices` 配列における正解位置を表す。

アプリ側で `choices` をシャッフルする場合は、シャッフル後の配列に合わせて正解位置を再計算する。

### 画像パス

画像あり問題では、次のように入れる。

```json
"image": "images/counter_hashi.png"
```

画像なし問題では、次のようにする。

```json
"image": null
```

これは問題データではなく、作問ルール改善用のメモです。

## 11. 作問の進め方

このルールを使って、ページ単位またはバッチ単位で本番作問を進める。

### テスト対象候補

| ページ | 確認したい型 |
|---|---|
| p012 | `meaning_context` |
| p013 | `age_word` |
| p014 | `image_word` |
| p205 | `month_hint` + `subtype: month_word` |
| p217 or p218 | 特殊暗記系の扱い確認 |

### テスト時に見るポイント

- 青ちゃんにとって文が長すぎないか
- 選択肢が簡単すぎないか
- 選択肢が難しすぎないか
- 作問データ上で正解位置が極端に偏っていないか
- アプリ側でシャッフルする前提のJSONになっているか
- 解説がわかりやすいか
- 画像が必要な問題と不要な問題の境目が適切か
- JSONにしやすいか
- アプリ表示に向いているか
- `image_word` で必要な場合に `hint` が入っているか
- `manner_word` などの補助分類が必要な場合に `subtype` が入っているか

## 12. チェックリスト

作問後、各問題について次を確認する。

### 内容チェック

- [ ] 教材本文をそのまま使っていない
- [ ] 語句の意味が正しい
- [ ] 例文の使い方が自然
- [ ] 青ちゃんが状況を想像できる
- [ ] 小3が読める文量
- [ ] 正解が1つだけ
- [ ] 誤答が不自然すぎない
- [ ] 誤答が正解と近すぎない
- [ ] 解説が短くわかりやすい
- [ ] ニュアンスがある語は必要に応じて補足している

### 選択肢チェック

- [ ] 選択肢が4つある
- [ ] 作問データ上で正解位置が極端に偏っていない
- [ ] アプリ側でシャッフルする前提になっている
- [ ] 正解だけが長すぎない
- [ ] 表現の粒度がそろっている

### JSONチェック

- [ ] `id` が一意
- [ ] `sourcePage` が正しい
- [ ] `type` が正しい
- [ ] `choices` が4つ
- [ ] `answerIndex` が0〜3
- [ ] `answerIndex` と元の `choices` 内の正解位置が一致している
- [ ] 画像なしは `image: null`
- [ ] 画像ありは画像パスが入っている
- [ ] `hint` がある場合、答えを直接言いすぎていない
- [ ] `subtype` がある場合、typeとの関係が自然
- [ ] `subtype: "onomatopoeia_word"` の場合、音・光・動き・気持ちなどの説明が明確
- [ ] `subtype: "clothing_word"` の場合、`hint` に見る場所や特徴が入っている
- [ ] `subtype: "clothing_word"` の画像ファイル名が `images/clothing_*.png` 形式になっている
- [ ] `subtype: "animal_idiom_word"` の場合、解説でたとえであることが分かるようにしている
- [ ] `subtype: "period_word"` の場合、例文や解説で「いつごろか」「どの期間か」が具体的に示されている
- [ ] `subtype: "weather_word"` の場合、`hint` に雨の強さ・季節・時間帯・続き方が入っている
- [ ] `subtype: "weather_word"` の画像ファイル名が `images/weather_*.png` 形式になっている
- [ ] `subtype: "day_time_word"` の場合、`hint` に空の明るさ・太陽の位置など注目ポイントが入っている
- [ ] `subtype: "body_part_word"` の場合、`hint` に体のどこかを示す短い説明が入っている
- [ ] `subtype: "body_part_word"` の画像ファイル名が `images/body_*.png` 形式になっている
- [ ] `subtype: "plant_idiom_word"` の場合、解説でたとえや慣用句としての意味が分かるようにしている
- [ ] `subtype: "house_part_word"` の場合、`hint` に家のどの部分かを示す説明が入っている
- [ ] `subtype: "house_part_word"` の画像ファイル名が `images/house_*.png` 形式になっている
- [ ] `subtype: "cooking_word"` の場合、`hint` に料理の動作が分かる注目ポイントが入っている
- [ ] `subtype: "cooking_word"` の画像ファイル名が `images/cooking_*.png` 形式になっている
- [ ] `subtype: "old_life_tool_word"` の場合、`hint` に道具の形や使い方の注目ポイントが入っている
- [ ] `subtype: "old_life_tool_word"` の画像ファイル名が `images/old_tool_*.png` 形式になっている
- [ ] `month` がある場合、1〜12の数値になっている
- [ ] `reading_context` で必要な場合、`answerReading` が入っている
- [ ] `counter_image` で必要な場合、`counterTarget` / `counter` / `counterReading` が入っている
- [ ] 月の異名の由来を断定しすぎていない
- [ ] 引用符が半角の `"` になっている

## 13. stable運用ルール

このファイルは正式版として、本番作問に使用します。

作問を進める中で改善点や追加ルールが出た場合は、このファイルを直接更新します。

### 更新のタイミング

- 新しいページの作問で新しいtypeやsubtypeが必要になったとき
- 判断に迷うケースが出て、ルールを明確にしたいとき
- 作問後に `ruleDraftRevisionNotes` に記録したメモを正式ルール化するとき

### 更新時のルール

- ファイル冒頭の「更新日」を書き換える
- 「今回の更新」に変更内容を箇条書きで追記する
- 変更箇所は上書きし、古いルールは残さない
